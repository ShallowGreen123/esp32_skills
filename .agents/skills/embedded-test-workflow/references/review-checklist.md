# 硬件审阅清单

在读取原理图和数据手册时，先抽取这些事实，再决定代码怎么写。

## 第一层：先确认硬件事实
- 模块名称、版本、主芯片型号
- 供电电压、工作电流、电平标准
- 使能脚、复位脚、唤醒脚的极性
- 中断脚是否存在，触发方式是什么
- 通讯接口类型：GPIO / UART / I2C / SPI / ADC / PWM / Wi-Fi / BLE
- 启动顺序、稳定时间、复位后等待时间
- 模块初始化命令、寄存器、握手流程
- 可观测输出：状态脚、串口回包、屏幕现象、网络状态、电压变化

## 第二层：转换成软件问题
把上面的硬件事实转换成这些软件实现要点：
- 需要定义哪些 GPIO
- `setup()` 中必须按什么顺序初始化
- 哪些动作必须加 `delay()` 或超时等待
- 结果要通过什么日志或现象来判定
- 是否需要先做复位、校准、握手

## 第三层：映射成 example
至少考虑这四类 example：
- 基础连通：模块是否上电、是否可初始化、是否有最小响应
- 功能演示：模块的主要卖点功能
- 异常处理：参数错误、设备未连接、外部条件不满足
- 稳定性验证：重复执行、长时间运行、边界条件

## 必须显式提醒的风险
- 上拉/下拉要求不清楚
- 高低电平有效性不清楚
- ESP32 启动相关 GPIO 被复用
- 模块和主控电平不匹配
- 上电时序或复位时序描述不完整
- 数据手册没有给出超时建议
- 用户口述和资料内容不一致
