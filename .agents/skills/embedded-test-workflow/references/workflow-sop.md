# 标准 SOP

这份 SOP 用于把嵌入式演示工程从“接到需求”推进到“固件可交付”。

## 1. 接收需求
- 输入：模块名称、功能目标、目标板卡、现有仓库
- 动作：写一句话目标，明确演示对象、成功标准、是否需要现场可见效果
- 输出：任务摘要
- 结束条件：知道要证明什么，不再停留在泛泛需求

## 2. 阅读硬件资料
- 输入：原理图、数据手册、引脚表
- 存放要求：统一整理到 `hardware/schematic`、`hardware/datasheet`、`hardware/pinmap`、`hardware/notes`
- 动作：按 [review-checklist.md](review-checklist.md) 提取关键信息
- 输出：引脚映射、初始化顺序、约束列表
- 结束条件：能说明代码应该如何初始化模块

## 3. 建立软件抽象
- 输入：硬件分析结果
- 动作：区分“公共逻辑”和“单个演示逻辑”
- 输出：
  - `demo_config.h`
  - `demo_pins.h`
  - 需要进入 `demo_core` 的公共能力列表
- 结束条件：公共层和 example 边界清楚

## 4. 拆分 example
- 输入：模块能力和演示目标
- 动作：按“一例一目标”拆分
- 输出：example 清单和每个 example 的成功判据
- 结束条件：没有一个 example 同时承担多个主要任务

## 5. 搭建工程骨架
- 输入：example 清单
- 动作：创建目录、公共头文件、`platformio.ini` 环境
- 输出：可编译骨架
- 结束条件：每个 example 都有独立入口

## 6. 实现演示代码
- 输入：骨架和硬件约束
- 动作：
  - 先实现基础 example
  - 再实现功能 example
  - 最后补异常和稳定性 example
- 输出：可运行源码
- 结束条件：串口日志足以判断成功和失败

## 7. 编译与验证
- 输入：源码和板卡
- 动作：
  - 编译每个 example
  - 烧录关键 example
  - 记录串口日志和结果
- 输出：验证记录
- 结束条件：至少关键 demo 可以稳定复现

## 8. 固件归档
- 输入：编译输出
- 动作：把 `.bin` 放入 `firmware/`，并附带版本、功能、日期信息
- 输出：可交付固件
- 结束条件：别人拿到固件和源码能知道对应哪个功能

## 9. 复盘沉淀
- 输入：本次开发过程中的问题和经验
- 动作：把通用模式沉淀到 `demo_core`、`scripts`、模板文件
- 输出：可复用资产
- 结束条件：下次同类模块不再从零开始
